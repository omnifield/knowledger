# Blueprint (ЧЕРНОВИК) — Omnifield Workspace Platform

> **Статус:** черновик на ревью user'а. Не канон, пока не «ок». Автор: architect-координатор, 2026-07-12.
> **Решение near-term (принято user):** строим на **голом docker-compose + devcontainer-спеке**. DevPod — слой портируемости на потом. K8s — escape-hatch на масштаб.

---

## 0. Что это и зачем

**omnifield = не папка, а платформа-воркспейс** (internal developer platform). Юзер получает **свою систему**: разворачивает воркспейс на любой тачке с докером, живёт в нём, создаёт в нём продукты. Каждый продукт изолирован (свой контейнер = канон «модули»), но всё объединено удобным **хабом**.

**Боль, которую лечим** (ловлена вживую 2026-07-12, ~30 мин на «просто начать работать»): разбросанные контейнеры + ручная проводка gateway/портов + ручной запуск серверов и агентов + разъезжающиеся креды. Причина — нет цельной модели, есть куча ручных швов.

---

## 1. Модель — сиблинги + хаб (БЕЗ docker-in-docker)

```
                 ХОСТ (любая тачка с Docker)
   ┌──────────────────────────────────────────────────────────┐
   │  docker-compose проект = ВЕСЬ воркспейс                     │
   │                                                            │
   │   ┌──────── ХАБ (front-door) ────────┐                     │
   │   │  gateway (nginx :8080)           │  единая дверь       │
   │   │  brainer (control-plane/пульт)   │  живая карта, спавн  │
   │   │  code-server (веб-IDE) [позже]   │  удобный вход        │
   │   └──────────────────────────────────┘                     │
   │            │ сеть воркспейса (сиблинги, не вложены)         │
   │   ┌────────┼─────────┬──────────────┬──────────────┐       │
   │  [weber]  [chater]  [brainer-svc]  [<новый продукт>] …      │
   │  контейнер контейнер   …            каждый в СВОЁМ           │
   └──────────────────────────────────────────────────────────┘
        ▲                                    ▲
        │ прямой доступ (docker exec/CLI)    │ удобный доступ (хаб)
        └──────────────── ЮЗЕР ──────────────┘
```

- **Продукты — сиблинги**, каждый в своём контейнере (изоляция, независимый lifecycle, prod-parity). **Не вложенные** (никакого docker-in-docker — лишняя тяжесть и хрупкость).
- **Хаб = обычный ВЕБ-АПП** (решение user): открывается в браузере по URL (brainer-фронт за gateway) — как любой сайт. Не родитель продуктов, а front-door рядом: `gateway` (единая дверь) + `brainer` (пульт/живая карта/спавн агентов). **`code-server` (веб-редактор кода) — НЕ «весь хаб», а его компонент** (открыть код продукта прямо в браузере).
- **Двойной доступ (канон-принцип):** юзер лезет в любой контейнер **напрямую** (`docker exec`/CLI) **ИЛИ** через хаб-веб-апп для максимального удобства. **Прямой путь всегда работает как fallback** — не запираем всё за хабом.
- **Весь воркспейс = один compose-проект** → `docker compose up` на любой тачке. Портируемо by construction. **Оркестрация (compose/gateway/генерация) — зона devopser** (он владеет docker/gateway/CI/deploy), НЕ отдельный репо.

## 2. Стек-слои — что БЕРЁМ готовое, что СТРОИМ (OPENSOURCE-FIRST)

| Слой | Берём готовое | Строим своё |
|---|---|---|
| Оркестрация сиблингов | **docker-compose** | генерация compose из манифестов (devopser) |
| Среда продукта | **devcontainer-спека** (`dockerComposeFile` + `service`) — читают И VS Code, И DevPod | шаблоны продуктов (devopser) |
| Единая дверь | **nginx** (gateway) | маршруты из манифестов (devopser) |
| IDE-поверхность | **VS Code** (Reopen in Container) сейчас · **code-server** (веб) в хаб позже | — |
| Портируемость + агент (порты/креды/логи) | **DevPod** — ОТЛОЖЕН (near-term без него) | — |
| **Хаб-интерфейс** (пульт, карта, спавн) | — | **brainer** (уникальное, наше ядро) |

**Ключ:** структурно ничего не изобретаем. `devcontainer.json` на продукт указывает на общий `docker-compose.yml` и выбирает свой сервис (`shutdownAction: none`, общая сеть) — это НАТИВНЫЙ способ выразить «продукт-контейнер в общем воркспейсе», и ровно то, что читает VS Code (твой IDE-пивот) и позже DevPod (портируемость).

## 3. Два слоя: продукт самодостаточен, хаб знает только необходимое (уточнение user)

**Продукт = самодостаточный контейнер.** Несёт СВОЮ полную конфигурацию (Dockerfile/compose-сервис, как собирается/запускается, dev-сервисы, все настройки) и поднимается САМ — `docker compose up` только этого продукта, на любом сервере под конкретную цель. **Хаб для этого НЕ нужен** (канон самодостаточности: продукт не зависит от хаба).

**Тонкий манифест — интерфейс продукт↔хаб.** Продукт выставляет МАЛЕНЬКУЮ визитку ровно с тем, что хабу НАДО знать чтобы интегрировать, не больше:
- identity: `name` · `type`
- как достучаться: `gatewayRoute` · `port(s)`
- интеграция: `scopes[]` · `spawnEligible` · `deps[]` (на другие продукты, если есть)

**Расширенная инфа и настройки остаются ВНУТРИ продукта** — хаб их не видит и не хранит.

Роли слоёв (аггрегация, НЕ генерация внутренностей):
- **Продукт** владеет своей runnable-конфигурацией (compose-сервис/Dockerfile + dev-сервисы) — самодостаточен, запускается один.
- **devopser (оркестрация)** АГРЕГИРУЕТ compose-определения продуктов + проводит gateway/порты по тонким манифестам → рабочий воркспейс. Склеивает самодостаточные части, а не генерит внутренности продукта.
- **Хаб (brainer)** читает тонкие манифесты → живая карта, маршрутизация, спавн. Опциональный слой удобства поверх.

Тонкий манифест поглощает хардкод `BRAINER_REPOS` (brainer `config.py`) и devopser `registry/products.md` (C10) — но как **тонкий индекс интеграции**, НЕ как хранилище всех настроек продукта.

## 5. Креды и identity

- **Сейчас:** один стор (volume `omnifield-secrets`), юзер работает ТОЛЬКО в Docker → нет расхождения host↔volume (корень сегодняшней auth-боли). Onboarding-seed `.claude.json` печётся один раз (devopser B8).
- **Позже:** DevPod credential-forwarding — форвардит креды из одного клиент-источника в контейнер, без копии в volume вообще.
- **Автономный флот** (пульт плодит агентов): опция `ANTHROPIC_API_KEY` — если/когда упрёмся в rate-limit подписки или захотим per-token видимость. Не сейчас (решение user).
- **Identity:** `OMNIFIELD_SCOPE` → роль (main=architect, иначе owner). Дом **workspace-архитектора** = хаб-контейнер (снимает вопрос «где живёт координатор» — не костыль /workspaces, а штатный хаб).

## 7. Роль-тир (вложенный)

- **workspace-scope:** workspace-архитектор (+ будущие marketing/manager/…) — с ним user обсуждает верхнеуровневое (продукты, роадмапы). Живёт в хаб-контейнере.
- **product-scope:** product/frontend/backend-архитекторы + owners — исполняют внутри продукт-контейнеров.
- Ложится на roles=data/flow-пресеты (ADR 035) + brainer-харнесс.

---

## 8. Guardrails (не-цели)

- ❌ docker-in-docker (сиблинги, не вложенность).
- ❌ свой VM/оркестрационный слой — берём compose/devcontainer/DevPod.
- ❌ запирать всё за хабом — прямой доступ всегда работает.
- ✅ **продукт запускается БЕЗ хаба** (`docker compose up` сам по себе, на любом сервере под свою цель) — хаб опционален, не зависимость. Манифест тонкий; расширенные настройки живут ВНУТРИ продукта.
- ✅ blueprint-before-code на каждый инкремент; homework-before-ADR на vendor-механику.
- ✅ air-gapped-дружелюбие — проверить offline-поведение (кэш образов) на инкременте 5 (DevPod).

## 9. Решения по открытым вопросам (ревью user 2026-07-12)

1. **Форма манифеста** — ✅ **отдельный чистый манифест** (`omnifield.yaml` / имя финализирует knowledger), НЕ дописка в nx `project.json`: платформа не лезет в nx-конфиг, и не-JS продукты (Go/Python) `project.json` могут не иметь. Точную схему проектирует knowledger (инкремент 1).
2. **IDE-поверхность хаба** — ✅ **хаб = обычный веб-апп в браузере** (brainer-фронт). code-server = компонент хаба (веб-редактор кода продукта), не отдельная поверхность. Локальный VS Code Reopen-in-Container остаётся как прямой путь.
3. **Compose-топология** — ✅ **ГЕНЕРИМ, не пишем руками**: продукт декларирует только манифест, devopser из всех манифестов собирает рабочий compose (инкремент 2). Источник = per-product (манифест), запуск = один сгенерированный compose. Снимает и merge-конфликты общего файла, и ручную склейку.
4. **Где живёт оркестрация** — ✅ **в devopser** (его зона: docker/gateway/CI/deploy). Отдельный репо `omnifield/workspace` НЕ создаём.
