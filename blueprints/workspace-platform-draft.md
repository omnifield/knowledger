# Blueprint — Omnifield Workspace Platform (v2)

> **Статус:** живой канон модели. База зафиксирована пачками решений user 2026-07-12/13 (ниже
> отражены целиком). Модель ещё дополняется по ходу, но от зафиксированного НЕ отходим.
> **Стек near-term (user):** голый docker-compose + devcontainer-спека. DevPod — портируемость на потом.
> K8s — escape-hatch на масштаб.

---

## 0. Что это

**omnifield = не папка, а платформа-воркспейс** (internal developer platform). Юзер с одним лишь
Docker на машине ставит omnifield и получает **свою систему**: разворачивает воркспейс, живёт в нём,
создаёт продукты. Каждый продукт изолирован (свой контейнер), всё объединено **хабом**.

**Дев-флоу юзера == наш дев-флоу (догфуд).** Мы строим weber/brainer ровно так, как юзер строит своё.
Одна система, одинаковые технические права (§7).

---

## 1. Хаб — ОДИН на машину (не per-воркспейс)

**Хаб = тонкий менеджер воркспейса, как nx для монорепо. НЕ продукт.** Один на всю машину, сканит
ВСЕ контейнеры — и тулзы, и проекты. Суть ядра:
- **Реестр** — инфа обо всех контейнерах (из манифестов, §4);
- **Канал** — общение с контейнерами (управление + точечный доступ); старт — Portainer, свой — позже;
- **Дверь :8080** — единый вход (gateway) + лендинг.

⚠️ **brainer и devopser — ПРОДУКТЫ, не хаб.** brainer = продукт «агенты» (пульт/спавн), devopser =
продукт «девопс» (skeleton/CI/генерация конфигов). Их капабилити **втягивается В хаб** по надобности;
сам хаб продукт-агностичен. База хаба уже есть: gateway :8080 + лендинг.

---

## 2. Воркспейс = группировка на уровне ИНТЕРФЕЙСА хаба

- **Воркспейс — НЕ контейнер и НЕ отдельный хаб.** Это логическая группировка «какие контейнеры
  вместе» в интерфейсе одного хаба. Юзер внутри хаба создаёт воркспейс → в нём проекты → проекты
  юзают тулзы.
- Группировка **реальная** (не косметика) и служит **юнитом деплоя** (прод-флоу §8: воркспейс →
  продукты → на VPS/стор).
- **По дефолту ≠ сетевая изоляция** (разделение интерфейсное; контейнеры плоские сиблинги на общей
  docker-сети). **Изоляция — опционально**, через тул-группировщик (нативные docker networks) — когда
  надо. Мульти-тенант/прод-изоляция — проработка позже.

---

## 3. Продукты — сиблинги, самодостаточны, у каждого ЛИЦО

- **Сиблинги**, каждый в своём контейнере (изоляция, независимый lifecycle, prod-parity). **Без
  docker-in-docker** (не вложенность — лишняя тяжесть/хрупкость).
- **Самодостаточен:** несёт свою полную runnable-конфигурацию (Dockerfile/compose-сервис, dev-сервисы,
  настройки) и поднимается САМ — `docker compose up` только его, без хаба. Хаб опционален, не зависимость.
- **У каждого продукта есть ЛИЦО** (часть шаблона продукта, devopser-skeleton):
  - интерактивный (brainer) → лицо = интерфейс/пульт;
  - пакетный (weber — чистые пакеты, юзеру тыкать нечего) → лицо = **описание пакетов**.
  - weber в хабе с лицом-описанием; его sandbox-app'ы = внутренние тест-полигоны, НЕ лицо.

---

## 4. Тонкий манифест — интерфейс продукт↔хаб

- **`omnifield.yaml`** в корне репо каждого продукта (JS и не-JS одинаково). Тонкая **визитка**:
  identity (`name`·`type`) + reach (`routes`: путь+внутр.порт) + integration (`scopes`·`spawnEligible`·`deps`).
- Расширенные настройки/lifecycle (`devbox.services.json`, Dockerfile, секреты) — **ВНУТРИ продукта**,
  в манифест не идут. `docker compose up` продукта манифест не читает (самодостаточность).
- **Контракт `@omnifield/contract-manifest`** (инк1, ✅ собран): Zod-схема `.strict()` — источник правды;
  из неё эмитится `omnifield.schema.json` (кросс-язычно, не-JS валидируют им). Ноль дрейфа.
- **`registry/products.md` и `BRAINER_REPOS` ретайрятся** — выводятся сканом манифестов, не источник.

---

## 5. Интерфейс хаба

**Существительные (навигация) + глаголы (капабилити):**
- **Существительные — категории:** **Тулзы · Воркспейсы**. Каждый item → превью → его **лицо**.
- **Глаголы — капабилити на item'е:** **DevOps** (управление контейнерами = Канал/Portainer),
  **Develop** (VS Code открывает репо продукта/воркспейса), **Agents** (brainer). Даёт втянутый тул,
  хаб — поверхность.
- Доступны на **двух уровнях:** продукт-уровень (в рамках одного самодостаточного продукта) и
  **воркспейс-уровень** (пульты НАД продуктами: workspace-devops, workspace-агенты).

---

## 6. Контролируемая гибкость (правь источники, не генерённое)

Юзеру дана полная сила через VS Code, но **база защищена гардрейлами**:
- **Правишь ИСТОЧНИКИ** (манифест `omnifield.yaml`) — гардрейл вшит: Zod `.strict()` режет кривизну.
- **Генерённое** (compose/nginx/лендинг) помечено `GENERATED`, руками не клобберишь — перегенерится.
- **Глубокая кастомизация = валидируемый override** (`docker-compose.override.yml` поверх базы) —
  базовые механы целы, твоё сверху. Полный **eject** (забрать генерённый compose себе) = крайний
  escape-hatch, не дефолт.
- VS Code = сырой редактор; контроль = валидация в devcontainer (schema + drift-check + husky-гейты).

---

## 7. Тул vs продукт, права, публикация

- **Тул vs продукт — НЕ технический тип.** Один артефакт (weber) = «тул» когда юзаешь, «продукт-в-
  разработке» когда строишь. Категория Тулзы/Воркспейсы — **организационный ярлык**, не права.
- **Права технически равны у всех** (чистый догфуд): и мы, и юзер и разрабатываем, и юзаем. Машинерию
  пермишенов сейчас НЕ строим.
- **Единственный гейт — ПУБЛИКАЦИЯ в общий дистрибутив** (сайт/маркет): чужие дополнения → конкурс +
  наше ревью; наше → напрямую. Это **отдельный будущий слой** (маркетплейс/контрибуция), НЕ ядро;
  локальная разработка свободна.
- **«Кто»-разделитель** (омнифилд-дев vs юзер) — будущий **identity-слой** для опциональных лимитов.
  Задел, не сейчас. Ложится на `OMNIFIELD_SCOPE`/роль-тир (ADR 035).

### Агенты — трёхъярусные пресеты
**Хаб-уровень** (высокоуровневая консультация) → **воркспейс** (архитектор + по агенту на продукт) →
**продукт** (владельцы). Топология — пресеты (базовый = гл. архитектор воркспейса + продуктовые).

---

## 8. Дев/прод флоу и дистрибуция

- **Дев-флоу == юзер-флоу** (догфуд, §0).
- **Прод-флоу:** воркспейс → накидал продуктов → дальше что угодно с контейнерами: деплой на VPS,
  собрать app в стор, или поднять **прод-хаб** для конечных юзеров.
- **Прод-хаб** = каталог продуктов для конечных юзеров (тот же механизм лендинг-из-манифестов, другая
  аудитория/капабилити, чем дев-хаб).
- **Дистрибуция:** юзер качает с сайта **хаб-бутстрап** (НЕ монолитный образ — сиблинги, не
  docker-in-docker) с docker-доступом, он поднимает тулзы-сиблинги. **Пресеты:** «фронт» → weber;
  «фронт + конструктор» → weber + webstudio; и т.д. Хаб идёт автоматом, тулзы докачиваются в хабе.

## 9. Стек-слои (opensource-first)

| Слой | Берём готовое | Строим своё |
|---|---|---|
| Оркестрация сиблингов | **docker-compose** | генерация compose из манифестов (devopser hub-core) |
| Среда продукта | **devcontainer-спека** (VS Code + позже DevPod) | шаблоны продуктов + ЛИЦО (devopser-skeleton) |
| Единая дверь | **nginx** (gateway) | маршруты из манифестов (devopser hub-core) |
| Канал (docker-ops) | **Portainer** (старт) | свой control-plane (позже — Portainer не покроет спавн-агентов/кастом) |
| Редактор | **VS Code** (Reopen in Container); code-server (веб) позже | — |
| Портируемость | **DevPod** — отложен | — |
| Реестр + лицо + агенты | — | **hub-core** (devopser) + **brainer** (агенты) — наше ядро |

## 12. Guardrails (не-цели)

- ❌ docker-in-docker (сиблинги, не вложенность).
- ❌ свой VM/оркестрационный слой — берём compose/devcontainer/DevPod/Portainer.
- ❌ запирать всё за хабом — прямой доступ (docker exec/CLI) всегда работает.
- ❌ переписывать адреса втихую в gateway — продукт отдаёт по декларированному пути (без rewrite-шимов).
- ❌ полумеры/костыли — строим на источниках (манифесты), не на рукописных интеримах.
- ✅ продукт запускается БЕЗ хаба; хаб опционален.
- ✅ права технически равны; гейт только на публикации.
- ✅ blueprint-before-code на инкремент; homework-before-ADR на vendor-механику.
- ✅ air-gapped-дружелюбие (offline-кэш образов) — проверить на DevPod.
