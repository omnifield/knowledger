# Containers-only — исполнение только в контейнере

Верхний закон — `../../POLICY.md`. Родственные: [dogfood-product-flow](dogfood-product-flow.md)
(наш флоу = флоу юзера — этот принцип его прямое следствие),
[ecosystem-self-sufficiency](ecosystem-self-sufficiency.md). Операционка —
devopser `briefs/docker-dev-environment.md` (devbox, амендмент 2026-07-10).

## Тезис

**Всё, что запускается в экосистеме, работает только в контейнере.** Нет Docker на
машине — нет работы. Хост-путь не проектируем и не поддерживаем: у нас нет ресурса
держать два пути, а prod-parity требует, чтобы наш способ запуска был буквально
способом юзера.

## Правила

1. **Исполнение — в контейнере**: тулчейн (devbox), сервисы (compose), прогоны агентов —
   всё в докере. Единственный prerequisite машины: git + Docker Engine + Claude Code.
2. **Файлы — на машине**: рабочие копии живут на хосте (на Windows — в WSL2 FS),
   контейнер монтирует ТОЛЬКО workspace (bind-mount). Изоляция хоста — охватом маунта.
3. **Docker Desktop — не требование**: стандарт = Docker Engine (Windows — в WSL2;
   серверы — engine). Desktop допустим как личный выбор, не референс.
4. **Версии не в образе**: контейнер — оболочка тулчейна; версии-исполнители остаются
   пинами репо (toolchain-pins). Обновление образа ≠ смена тулчейна.
5. Исключения (если жизнь заставит) — только явным решением product owner'а с записью;
   «на голой тачке быстрее» аргументом не является.

## Why

Раздвоение хост/контейнер = двойная стоимость каждой инструкции, каждого гайда, каждой
отладки «а у меня не так». Контейнер делает окружение воспроизводимым байт-в-байт
(machine = cattle доведён до конца) и совпадающим у нас, у юзера и в CI. Файлы на хосте
сохраняют владение данными и привычную работу с IDE/бэкапами.
