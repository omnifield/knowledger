# Модель компонентов

БАЗА. Читать ДО любой UI-работы. Любая визуальная композиция — СТРОГО в kit пресетом, никогда в пакете/аппе.

## Тезис

- **Композиция — только в kit, пресетом.** Не в пакете, не в аппе. Потребитель выбирает пресет, не строит лайаут.
- **Карточка = сущность.** Рисует данные по ключам; вид = **пресет**, полнота = **слоты** (вкл/выкл), точечно = **оверрайд пропсами** — НЕ ручной лайаут.
- **Список = пресеты форм отрисовки.** Элементы = любые компоненты.
- **Свои данные обязаны рисоваться своими компонентами.** Сущность домена → её карточка/шейп, не сырой div.

## Карточка (сущностный режим)

Карточка получает данные и рисует их **по ключам-слотам** (title/subtitle/badge/tags/meta/…). Три рычага настройки, строго в этом порядке:

1. **Пресет** — именованный вид (`word-compact`, `word-full`, `entity-meta`). Меняет форму целиком.
2. **Слоты** — включить/выключить части (нет definition → слот выключен). Полнота карточки.
3. **Оверрайд пропсами** — точечная правка отдельного места.

Ручной лайаут внутри потребителя (div-вёрстка карточки в аппе/пакете) — **нарушение**. a11y (interactive/onClick→role/tabIndex/Enter-Space) вшита в компонент, не в потребителя.

## Список

Список = набор **пресетов форм** (плоский/секционный/аккордеон-группы). Что рисовать элементом — любой компонент (в т.ч. сущностная карточка). Потребитель даёт данные + выбирает форму, не верстает.

## Почему

Композиция, размазанная по потребителям, = дивергенция (один и тот же тег рисуется по-разному в N местах) + невозможность прогнать через рендерер/стор studio. Композиция в kit пресетом = один источник вида, все потребители равны, studio-рендерер видит структуру. Связь с [registration](registration.md) (2 слоя) и [kit-first](kit-first.md) (ноль классов).
