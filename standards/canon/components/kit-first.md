# Kit-first — единый источник визуала

## Тезис

Весь интерфейс — из **kit** (`@omnifield/*` UI-kit). Единый UX всего продукта — свойство kit'а, не семейного пакета. Все потребители (apps / shell / boost / домены / app-host) — **равные**, ни у кого нет своего визуала.

## Три рычага

1. **Пресеты** — вид задаётся именованным пресетом в kit, не композицией у потребителя ([[component-model]]).
2. **Бан классов везде** — потребитель НЕ пишет raw-классы. Классы (Tailwind/CVA) живут **только внутри kit-компонента**. Потребитель управляет видом **только через props**.
3. **Kit-хуки** — переиспользуемое поведение (a11y, active-segment, dnd) отдаётся хуками из kit/по-dependency-пакетам, не копипастой.

## Правила

- **Потребитель — props-only.** Ноль `class`/`className` с версткой в аппе/пакете/шелле. Нужен вид, которого нет — расширяешь **kit** (owner-флоу), не обходишь классом.
- **Ноль raw-style.** Никаких `<style>`-блоков и inline-стилей в потребителе; состояние — в JS + токены ([[tokens]]).
- **Tailwind сканит только kit.** Источник классов — папки kit. Потребители вне scan-таргета → класс у потребителя просто не сгенерится (structural-защита).
- **Недостающий примитив** (иконки, mono-типографика и т.п.) заводится **в kit**, не хендроллится у потребителя. Сначала существующая либа (Kobalte и т.п.), потом своё.
- **app-host пакеты = данные + композиция**, ноль UI/классов.

## Почему

Классы у потребителей = визуальная дивергенция + невозможность единого рескина + Tailwind-scan-хаос. Единый источник (kit) + props-only граница = один UX, рескин из одной точки, studio-рендерер видит только декларативные props. Продуктовый слой, не семейный ([[../principles/modules-no-crutches]]).
