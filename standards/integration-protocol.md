# Протокол интеграций — внешние скоупы

Верхняя формулировка — `POLICY.md §5`. Стандарт для **внешнего скоупа** — человека/команды/сервиса вне owner'ов репо: предметный эксперт (учитель-контент), сторонний разработчик (движок/клиент), внешний сервис (чужой backend). Интеграция идёт по стандарту, не ad-hoc «переброс наугад».

## Три столба

1. **Онбординг** — что скоуп читает на входе, что от него ждут, где границы.
2. **Версионируемый контракт** — шов через `commons/contracts` (typed, версия = точка синхронизации), НЕ прямой импорт внутренностей соседа.
3. **Канал** — явное место общения + протокол обмена + один liaison.

## 1. Онбординг

- **Charter-документ** скоупа: зона ответственности, что читает, границы, канал, liaison. Один файл, single source of truth про скоуп.
- Скоуп читает **`POLICY.md` + релевантные `canon/`-темы** (не весь канон — только его касающееся).
- Эскалация: `скоуп → liaison (architect репо) → product owner`. Снизу вверх ([[canon/packages/ownership]]).

## 2. Контракт как шов

- Cross-repo / cross-scope концерн → **контракт в `commons/contracts`** ([[canon/principles/modules-no-crutches]]). Скоуп общается через опубликованный контракт, не через внутренности.
- **Версия контракта = видимый брейк** и точка синхронизации независимых сторон. Изменение = version-bump, не тихий дрейф.
- **DoR/DoD стадий применимы** ([[workflow/definition-of-ready]]): контракт Done (опубликован + версия зафиксирована) → downstream-скоуп стартует. Не «пишем наугад, пока контракта нет».
- Внешние ресурсы (URL/веса/ключи) — **user-supplied, не хардкодить** (self-host-first: всё через свой origin, ресурсы даёт потребитель).

## 3. Канал + liaison

**Один liaison на скоуп** (architect репо или назначенный owner). Общение — через liaison, не напрямую в код скоупа.

Протокол обмена (канон liaison, вычитан из практики):
- **Инструкция — ТОЛЬКО с копипаст-образцом.** Не «сделай по правилам X», а готовый пример, который скоуп копирует.
- **Правила не анонсировать до кода** — сначала рабочий образец/механика, потом формализация, не наоборот.
- **Спека = authoring-contract** — формат/шаблон, по которому скоуп производит контент/код; рассинхрон формата = поломка ингеста.
- **Чужой контент без явного «ок» не трогать** — материал скоупа (vault, дизайн, схемы) правится только с его подтверждения.
- **Один источник правды на тему** в канале, старое помечать `superseded` ([[workflow/docs-hygiene]]).

## Типы внешних скоупов (прецеденты)

| Тип | Канал | Контракт |
|---|---|---|
| **Предметный эксперт** (контент) | docs-папка канала, liaison через product owner | authoring-contract (формат/шаблон) + ингест-гейт (валидация на въезде) |
| **Сторонний разработчик** (движок/клиент) | onboarding-док + Q&A-тред | версионируемый JSON/API-контракт (мост); все вызовы через наш origin |
| **Внешний сервис** (чужой backend) | API-charter | typed endpoint-контракт в `commons/contracts` |

## Принцип

Внешний скоуп — не исключение из канона, а его частный случай: **та же граница модульности** ([[canon/principles/modules-no-crutches]]), только сторона шва живёт вне репо. Дисциплина (онбординг + контракт + канал) заменяет «доверие на слово» на видимый, версионируемый шов.
