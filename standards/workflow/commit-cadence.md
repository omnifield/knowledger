# Commit cadence — этапность + pre-commit гейт

Ритм разработки, общий для всех репо. Scope коммита — в `git-scope.md`; здесь — **когда** коммитим и **что зелёное** перед этим.

## Этапный ритм

Разработка идёт **по этапам**. Цикл каждого этапа:

```
этап → проверка → (ok) → commit → следующий этап
                  ↘ (not ok) → чиним, не коммитим
```

- **Не коммитим непроверенный этап.** Коммит = зафиксированное-рабочее, не «пусть полежит».
- **Проверка** = автотесты зелёные + (для UI) проверка в браузере product owner'ом. См. DoD «Тесты» в `definition-of-ready.md`.
- Непроверенные этапы **не батчатся** в один коммит.

## Pre-commit гейт (жёстче pre-PR)

Перед **КАЖДЫМ** коммитом локально зелёные:

- [ ] **Тесты** (unit + затронутая механика).
- [ ] **Линтеры** (formatter + правила).
- [ ] **Билд** затронутых пакетов.

Цель — исключить «локально ок, а пайплайн падает на линтере/билде». CI не должен краснеть на том, что ловится локально; красный CI постфактум = compound-долг.

## PR

PR открываем: **по завершению функционала** ИЛИ **по прямому указанию** product owner'а. Не на каждый коммит — коммиты накапливаются в ветке по этапам, PR = логическая единица (форма изменения, `git-scope.md`).

## Связь с эталон-гейтом

Pre-commit гейт ⊂ DoD, но мельче: DoD закрывает **item** (код+тесты+трейсы+доки+раскладка), pre-commit гейт держит зелёным **каждый этап** внутри item. Проходить pre-commit ≠ item Done.
