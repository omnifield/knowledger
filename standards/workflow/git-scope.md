# Git scope — scope = форма изменения

Scope коммита/PR определяется **логической формой изменения**, не авторством (порт оракул §0.1, расширен на cross-repo).

| Изменение | Scope |
|---|---|
| Локально в одном пакете/сервисе (фича/рефактор/тесты в owner-зоне) | **Per-package.** Owner делает PR, мержит после CI. Чистая releasability. |
| Cross-package в одном репо (одна логич. цепочка через несколько зон) | **Per-feature.** Один PR со скоупом фичи; координирует architect репо; owner'ы работают по своим файлам в этом PR. |
| Cross-repo (фича через FE↔BE↔devops) | **Через контракт.** Каждый репо — свой PR; связь = версия `@omnifield/contract-*`, видимая всем. НЕ переброс брифами через координатора. |
| Mixed WIP «куча всего» | Аномалия. Лечится одноразовым cleanup-PR. Не норма. |

## Инварианты

- **Scope ≠ авторство.** Owner коммитит ВСЁ, что в его scope попало, независимо кто строки писал.
- **Cross-repo фича НЕ фрагментируется post-factum.** Контракт связывает независимые PR в одну логическую единицу — версия контракта = точка синхронизации.
- **Pre-эталон vs post-эталон.** До эталона cross-repo/cross-package PR частые = симптом неготовности границ (не нарушение). После — редкие: либо contract change (легитимно), либо canon-violation (фиксим причину).

## Механика (per-repo)

Git hard-gate (какие операции режутся у subagent'ов), ветки, destructive-ops, conventional-commit формат — задаются **в каждом репо** (settings/hooks + agent-шаблоны `../agents/`). Здесь — только правило скоупа, общее для всех.
